## Day 1 — Foundation, Schema & Server Auth

**Goal: schema migrated, Express running with auth middleware, protected routes return 200/401. No client work today.**

The schema is the most expensive thing to get wrong — a bad migration costs hours to undo. Lock it in first before writing any application code.

### Server

```
server/prisma/schema.prisma
server/prisma/migrations/
```

**Schema models currently implemented:**

| Model | Fields |
|---|---|
| `User` | `id` (String, PK) · `email` (unique) · `username` · `avatarUrl` · `createdAt` · `updatedAt` |
| `Conversation` | `id` (String, PK) · `type` (enum: DM, CHANNEL) · `isPrivate` · `name` (nullable) · `workspaceId` (nullable) · `dmPair` (String?, unique) · `createdAt` · `updatedAt` |
| `ConversationMember` | `id` (String, PK) · `conversationId` (FK) · `userId` (FK) · `lastReadMessageId` (String?, FK → Message.id) · `joinedAt` |
| `Message` | `id` (String, PK) · `content` · `conversationId` (FK) · `userId` (FK) · `isEdited` (Boolean) · `createdAt` · `updatedAt` |

**Indexes and constraints to enforce in schema:**

- `ConversationMember`: `@@unique([conversationId, userId])` — prevents duplicate memberships
- `ConversationMember`: `@@index([userId, conversationId])` — required for sidebar/inbox query
- `Message`: `@@index([conversationId, id])` — message pagination ordered by UUIDv7 id
- `Conversation.dmPair`: `@unique` — enforces one-DM-per-user-pair at DB level

> ⚠️ **UUIDv7 generation:** All `id` fields must be generated using `import { uuidv7 } from 'uuidv7'` in the application layer before every `prisma.*.create()` call. Do **not** use `crypto.randomUUID()` — that produces UUIDv4 which is not monotonically ordered. Ordering and cursor comparison break silently if the wrong generator is used.

```
server/prisma/seed.ts                  ← looks up existing alice@example.com and bob@example.com,
                                          then upserts 1 DM conversation + 3 messages
                                          with conversation/member/message IDs generated by uuidv7();
                                          current dmPair format is userId_userId

server/src/lib/db.ts                      ← Prisma client using @prisma/adapter-pg

server/src/types/shared.ts             ← AuthRequest extends Express Request with req.user

server/src/middlewares/auth.ts         ← JWT validation via Supabase JWKS; no DB user sync here
server/src/middlewares/errorHandler.ts ← global error handler, consistent error shape
server/src/middlewares/rateLimit.ts    ← not implemented yet; when added, export
                                          generalLimiter and messageLimiter using
                                          express-rate-limit, then wire globally/API-specific

server/prisma/SUPABASE_QUERIES.sql     ← Supabase trigger syncs new auth.users rows into public.User

GET /api/me                            ← current user endpoint; /api/auth/me module is not required

server/src/app.ts                      ← express app config, in this order:
                                          1. CORS (origins: client URL, credentials: true)
                                          2. JSON body parser
                                          3. route-level auth middleware on protected API routes
                                          4. route registration
                                          5. errorHandler (last)

server/src/server.ts                   ← http.createServer(app), socket.io attach, listen
```

### End of Day 1 Checks

- [ ] `npx prisma migrate dev` runs clean, all tables visible in Supabase
- [ ] `npx prisma db seed` runs without errors
- [ ] Seed script requires existing alice@example.com and bob@example.com users, then upserts 1 DM conversation + 3 messages with UUIDv7 IDs
